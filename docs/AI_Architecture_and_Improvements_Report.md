# Feder AI Architecture and Efficiency Report

## 1. Introduction

This report provides a comprehensive overview of how Artificial Intelligence (AI) is integrated into the Feder application. It explores the underlying architecture, the specific functionalities implemented, the integration points within the React components, and detailed recommendations for improving performance, user experience, and overall efficiency.

## 2. Current AI Architecture

The AI module in Feder is primarily a front-end driven implementation designed to provide writing assistance directly within the `Editor` component. The system supports multiple Language Model (LLM) providers to offer flexibility for the user.

### 2.1 Supported Providers
The application natively supports three backend AI providers:
- **Google Gemini**: The default provider. Uses the `generativelanguage.googleapis.com` REST API. Models include `gemini-2.5-flash`, `gemini-2.5-flash-lite`, and `gemini-2.5-pro`.
- **OpenAI**: Connects to the traditional `api.openai.com/v1/chat/completions` endpoint. Supports models like `gpt-4o-mini` and `gpt-4o`.
- **Ollama (Local)**: Allows users to run local, privacy-first models (e.g., Llama 3) via a local server (defaulting to `http://localhost:11434`).

### 2.2 Global State and Configuration
The configuration for AI forms part of the global application settings. Access is managed primarily via `SettingsModal.jsx`. 
Values are nested under the `settings.ai` object, which acts as the single source of truth. It stores:
- `enabled`: Boolean toggle for the entire AI system.
- `provider`: Active provider choice.
- API keys, models, base URLs mapped to their specific providers.
- `triggerMode`: 'automatic' or 'manual' (Ctrl+Space).
- `maxWords`, `debounceMs`, `temperature`.
- `improvements`: Option to configure a *different* provider/model specifically for the text improvement feature versus the standard inline completion.

### 2.3 Core Capabilities

#### A. Inline Suggestions (Ghost Text)
This feature seamlessly predicts and continues the text the user is typing.
- **Trigger**: In **manual mode**, invoked by pressing `Ctrl+Space` or clicking the 'Sparkles' icon in the toolbar. In **automatic mode**, a debounced timer (default 1.5s) queries the AI whenever typing pauses.
- **Data Flow**: Handled inside `Editor.jsx`. When triggered:
  - The application grabs the context (up to 3000 characters before the cursor, 1000 characters after).
  - An `AbortController` handles edge cases (e.g., user starts typing again before the request resolves, cancelling the pending fetch).
  - Dispatches to `requestInlineSuggestion()` in `src/utils/aiSuggestions.js`.
  - Normalizes the generated text (stripping artifacts like newlines or leading ellipses).
  - Renders the proposition over the `Editor` textarea using a hidden `ghostRef` overlay component.
  - User hit `Tab` to accept and inject into the document.

#### B. Text Improvements
Selected text can be completely re-written for different styles.
- **Trigger**: Highlighting text in the Editor reveals a floating Action Widget. Clicking *Improve* opens a submenu: Formality, Coherence, Longer, Shorter.
- **Data Flow**: 
  - An action calls `handleRequestImprovement()` in the root `App.jsx`.
  - Dispatches to `requestTextImprovement()` which builds a specialized system prompt depending on the requested style (e.g., "You are a professional editor. Rewrite the text to be more formal...").
  - On return, the improved text is displayed (presumably in a modal comparing the original and newly proposed text).
  - User can decide to apply the changes, overriding their selection in the document.

## 3. Identified Bottlenecks and Potential Improvements

Currently, the AI architecture performs well for an MVP or standard desktop app, but several underlying elements prevent the system from being highly efficient, scalable, and responsive. Here are recommendations categorized by impact.

### 3.1 Network and Responsiveness (High Priority)

**1. Implement Streaming Output (Server-Sent Events)**
- **Issue**: The current implement uses synchronous HTTP fetch requests. For a completion of 30 words, the user must wait for the *entire* sequence to be generated by the API before they see anything on screen.
- **Improvement**: Switch to streaming endpoints.
  - Gemini: Use the streaming API or the Google Generative AI SDK, parsing chunks as they arrive.
  - OpenAI / Ollama: Enable `stream: true` in the request body. 
  - Update the `setSuggestion` state in React progressively. Ghost text that types out word-by-word greatly reduces perceived latency and prevents user frustration.

**2. Improved Debouncing & Trigger Heuristics**
- **Issue**: Automatic trigger is purely time-based. If set to 1.5 seconds, pausing mid-word triggers an API call yielding incoherent continuations.
- **Improvement**: Combine debouncing with semantic heuristics.
  - Only fire if the cursor is preceeded by whitespace, punctuation (e.g., `. `, `, `), or a newline. 
  - Avoid triggering in the middle of a continuous word string, drastically reducing redundant, expensive API calls and saving token costs.

### 3.2 Context Processing (Medium Priority)

**1. Semantic Context Window Management**
- **Issue**: Currently, `prefix.slice(-3000)` might slice the context abruptly in the middle of a sentence or a word 3000 characters prior, which can confuse strict LLMs.
- **Improvement**: Implement simple whitespace or newline boundary finding when slicing context. 

**2. Enhanced Editor Context injected as Metadata**
- **Issue**: The LLM acts purely on preceding/succeeding text, unaware of the *entire document's scope* (e.g., Title, Headings).
- **Improvement**: Pre-pend the system prompt with structural context. For example: `Document Title: {project.meta.title} | Current section heading: {...}`. This grounds the AI substantially, especially when context length is constrained.

### 3.3 State Management and Efficiency (Low-to-Medium Priority)

**1. Robust Caching**
- **Issue**: The Editor caches the last request via substring `prefix.slice(-100)|mode`. However, if you delete a character and re-type it, a new network request is triggered. 
- **Improvement**: Implement an LRU (Least Recently Used) cache for responses to store results for given context hashes.

**2. Token Limiter**
- **Issue**: The token estimate formula `maxTokens = Math.max(10, Math.ceil((maxWords || 12) * 2));` is basic. Often, different models have wildly varying token-to-word splits.
- **Improvement**: Use a lightweight tokenizer library (e.g., `tiktoken` equivalent for JS) to explicitly trim context and request outputs within strict, predictable budget constraints.

**3. Separation of Concerns**
- **Issue**: `aiSuggestions.js` directly implements API `fetch` requests coupled with parsing. 
- **Improvement**: Abstract this into a factory or Strategy design pattern (`GeminiProvider class`, `OpenAIProvider class`). This will make onboarding future models (like Anthropic Claude or Groq) much cleaner and easier to maintain.

## 4. Conclusion
The implementation of AI inside `Feder` offers a versatile integration encompassing the major models. By shifting to a **Streaming Response model** and **optimizing the auto-trigger event rules**, the perceived latency and efficiency of the application can be drastically improved with moderate developmental overhead.
